# MPU6050

## Summary

The MPU6050 is a combined 3-axis accelerometer and 3-axis gyroscope.

For chip-level reference, the useful topics are its address options, key registers, default sensitivity assumptions, and the temperature conversion formula.

## Datasheet

- Local file: [MPU-6000-Datasheet.pdf](../datasheets/MPU-6000-Datasheet.pdf)
- Original source: [TDK/InvenSense PDF](https://invensense.tdk.com/wp-content/uploads/2015/02/MPU-6000-Datasheet1.pdf)

## Facts We Rely On

- Default I2C address when `AD0` is low: `0x68`
- Alternate I2C address when `AD0` is high: `0x69`
- `PWR_MGMT_1` is register `0x6B`
- `SMPLRT_DIV` is register `0x19`
- `ACCEL_CONFIG` is register `0x1C`
- `GYRO_CONFIG` is register `0x1B`
- `WHO_AM_I` is register `0x75`
- Accelerometer data begins at register `0x3B`
- Temperature data uses the datasheet conversion:
  - `Temp C = raw / 340 + 36.53`
- At the default accelerometer full-scale range of `+-2g`, the sensitivity is `16384 LSB/g`
- At the default gyro full-scale range of `+-250 deg/s`, the sensitivity is `131 LSB/deg/s`

## Current Observed Configuration

This section is intentionally limited to one observed configuration example, not chip fact.

- Writing `0` to `PWR_MGMT_1` wakes the device from sleep.
- Writing `4` to `SMPLRT_DIV` gives a sample rate of `200 Hz` when the internal gyro output rate is `1 kHz`.
- Leaving `ACCEL_CONFIG` at `0 << 3` selects the default `+-2g` accelerometer full-scale range.
- If `GYRO_CONFIG` is left at its reset state, the gyro full-scale range remains the default `+-250 deg/s`.

## Known Quirks

- The chip exposes many configuration options, but this project currently uses only a small subset of them.
- Any firmware that changes gyro full-scale range must also change the scale conversion built on top of that range.

## Verified vs Inferred

Verified from code and datasheet:

- default address option `0x68`
- use of `PWR_MGMT_1`, `SMPLRT_DIV`, `ACCEL_CONFIG`, and `WHO_AM_I`
- temperature conversion basis
- default gyro sensitivity of `131 LSB/deg/s`

Inferred from current code behavior:

- that one current firmware path relies on the default `+-250 deg/s` gyro range because no explicit gyro-range write is present
